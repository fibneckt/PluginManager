## 1. 插件热加载/热卸载

### 现状

插件是编译进二进制的 Go 类型，通过 `init()` 注册。现有 `EnablePlugin`/`DisablePlugin` 可以运行时启停，但插件本身必须在编译时存在。

### 核心矛盾

真正的热加载意味着不重启进程就能加载新代码。Go 作为编译型语言（虽然有 `plugin.Open` 加载 `.so`，但限制多：相同 Go 版本、Linux only、依赖问题）。

### 推荐方案 — 外部进程协议

```
┌─────────────────────────────────────────────┐
│                 PluginManager                │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ Go 原生   │  │ Python   │  │ Node.js  │  │
│  │ plugin   │  │ 子进程    │  │ 子进程    │  │
│  │ (goroutine)│  │ os/exec  │  │ os/exec  │  │
│  └──────────┘  └──────────┘  └──────────┘  │
│       │            │ stdin/stdout │          │
│       │       JSON-RPC / JSON Lines         │
└─────────────────────────────────────────────┘
```

- **热加载** = `os.StartProcess` 启动插件进程，向 `PluginManager` 注册，分配 channel 和 worker goroutine
- **热卸载** = 关闭 stdin → 发 SIGTERM → 等超时 → SIGKILL → 清理 channel 和元数据
- Go 原生插件保留现有路径（`init()` 注册），外部插件走 `ExternalPlugin` 适配器
- 插件目录可被 `fsnotify` 监听，新增 `.so` / `.py` / `.js` 文件自动发现和加载

### 关键数据结构扩展

```go
type PluginRuntime string

const (
    RuntimeGo     PluginRuntime = "go"
    RuntimePython PluginRuntime = "python"
    RuntimeNode   PluginRuntime = "node"
    RuntimeBinary PluginRuntime = "binary"
)
```

## 2. 插件执行超时控制

### 现状

`handler/worker.go:safeRun()` 同步调用 `plugin.Run(input)`，没有超时机制。一个插件调用外部 API 卡住（比如 `plugins/ai.go` 的 `http.DefaultClient.Do(req)` 没有超时），整个 worker 就永久阻塞。

### 推荐方案 — context 超时 + goroutine 竞速

```go
// 在 worker.go 的 safeRun 中改造
func (m *PluginManager) safeRun(plugin model.Plugin, name string, data []byte) {
    // ... panic recovery 不变 ...

    timeout := m.getPluginTimeout(name) // 从配置读取，默认 30s
    ctx, cancel := context.WithTimeout(context.Background(), timeout)
    defer cancel()

    resultCh := make(chan model.PluginResult, 1)
    go func() {
        output, err := plugin.Run(input)       // 在子 goroutine 执行
        resultCh <- model.PluginResult{...}
    }()

    select {
    case result := <-resultCh:
        m.resultCh <- result
    case <-ctx.Done():
        m.resultCh <- model.PluginResult{
            Name: name,
            Err:  fmt.Errorf("plugin %s timeout after %v", name, timeout),
        }
        // 注意：纯 goroutine 超时无法真正杀死，goroutine 可能泄露
        // 进程隔离方案可以 SIGKILL 整个子进程，彻底清理
    }
}
```

### 要点

- 超时时间放在 `PluginConfig` 的 YAML 中，每个插件可配
- **goroutine 超时无法真正终止**（Go 的限制），被超时的 goroutine 会泄露。这是进程隔离（第3点）的另一个强动机 —— 子进程超时直接 SIGKILL，资源完全回收
- 短期方案：在现有 goroutine 模型下，超时后标记插件 errored，触发 worker 重建（stop old → start new）

## 3. 插件隔离方案

### 现状

所有插件共享同一个 Go 进程。一个插件的 panic（已有 recover）、死循环、内存泄漏会影响整个 Manager。

### 三层隔离，由浅入深

| 层级 | 机制 | 隔离效果 | 实现成本 |
|------|------|---------|---------|
| **Goroutine**（现有） | 独立 goroutine + recover | panic 隔离 | 零 |
| **进程** | `os/exec` 子进程 | 内存/CPU/崩溃隔离 | 中 |
| **容器** | Docker/podman | 文件系统/网络/完整隔离 | 高 |

### 推荐方案 — 进程隔离（最佳性价比）

```
PluginManager (主进程)
│
├── Go 原生插件 → goroutine（快速、低开销插件用这个）
│
├── 外部插件 A → os/exec → 子进程
│   ├── cgroups v2: memory.max = 256MB
│   ├── cgroups v2: cpu.max = 0.5 (半核)
│   └── 超时 → SIGKILL
│
└── 外部插件 B → os/exec → 子进程
    └── ... 同上
```

### 资源限制（Linux cgroups v2）

- 在 `/sys/fs/cgroup/plugin-<name>/` 下为每个插件创建子 cgroup
- 限制内存上限，超出 → OOM kill，进程终止，manager 收到信号
- 限制 CPU 配额，防止一个插件占满所有核心

### 进程崩溃处理

```go
// 子进程退出 → worker goroutine 检测到
go func() {
    err := cmd.Wait()
    m.handlePluginExit(name, err)
}()

func (m *PluginManager) handlePluginExit(name string, err error) {
    // 重启策略：指数退避，最多 3 次
    // 超过 → 标记 errored + 触发降级（第5点）
}
```

## 4. 插件依赖关系与版本约束

### 现状

插件有 `Name` 和 `Version`，但没有依赖概念。所有插件平级，启动顺序取决于 map 遍历（不确定）。

### 核心数据结构

```go
// model/info.go 扩展
type SemverConstraint string // 如 ">=1.0.0 <2.0.0", "^1.2.3"

type Dependency struct {
    Name       string           `json:"name" yaml:"name"`
    Constraint SemverConstraint `json:"constraint" yaml:"constraint"`
}

type PluginInfo struct {
    // ... 现有字段 ...
    Dependencies []Dependency `json:"dependencies,omitempty"`
    Optional     bool         `json:"optional,omitempty"` // 可选依赖?
}
```

### 版本约束语法（参考 npm/cargo）

- `^1.2.3` → `>=1.2.3 <2.0.0`
- `~1.2.3` → `>=1.2.3 <1.3.0`
- `>=1.0.0 <2.0.0` → 显式范围
- 使用 `github.com/Masterminds/semver/v3` 做解析和匹配

### 依赖解析流程

```
注册时验证:
  1. 收集所有依赖 → 检查依赖的插件是否存在
  2. 版本约束匹配 → 已注册插件版本是否满足约束
  3. 如果不满足 → 拒绝注册 或 标记为 "pending"（等待依赖满足）
  4. 循环检测 → DFS 检查依赖图是否有环

启动时排序:
  1. 构建有向无环图 (DAG)
  2. 拓扑排序 → 确定启动顺序
  3. 按序启动 worker → 确保依赖先就绪

运行时变更:
  - 插件 A 被禁用 → 检查哪些插件依赖 A
    → 如果 B 依赖 A 且非 optional → 级联禁用 B
    → 如果 B 依赖 A 且 optional → B 降级运行
  - 新插件注册 → 检查是否满足 pending 插件的依赖 → 自动启用
```

### 拓扑排序示例（Kahn 算法）

```
calculator (无依赖) → 先启动
ai (无依赖)         → 先启动
ai-plus (依赖 ai >=1.0.0, 依赖 calculator) → 等 ai 和 calculator 启动后启动
```

## 5. 插件执行结果的失败隔离与降级策略

### 现状

`safeRun` 中 `plugin.Run` 返回 error 后，插件被标记为 errored，但 `Enabled` map 中的条目不变，下次消息来还会重试。没有"连续失败 → 暂时熔断"的机制。

### 推荐方案 — 三层防护

#### 第一层：重试 + 退避

```go
type RetryPolicy struct {
    MaxRetries   int           // 最大重试次数（默认 3）
    BackoffBase  time.Duration // 退避基础时间（默认 100ms）
    BackoffMax   time.Duration // 退避上限（默认 10s）
    BackoffMult  float64       // 退避倍数（默认 2.0）
}

// 第1次失败 → 等 100ms 重试
// 第2次失败 → 等 200ms 重试
// 第3次失败 → 进入熔断
```

#### 第二层：熔断器（Circuit Breaker）

```
状态机:
  CLOSED ──连续N次失败──→ OPEN ──冷却时间到──→ HALF_OPEN ──成功──→ CLOSED
    │                       │                      │
    └──正常执行──→          └──快速失败──→          └──失败──→ OPEN
```

```go
type CircuitBreaker struct {
    FailureThreshold    int           // 连续失败 N 次打开（默认 5）
    CooldownPeriod      time.Duration // 冷却时间（默认 30s）
    HalfOpenMax         int           // 半开状态最多尝试次数（默认 1）
    state               cbState       // closed/open/half_open
    consecutiveFailures int
    lastFailureTime     time.Time
}
```

熔断打开后，对插件的请求立即返回 `ErrCircuitOpen`，不实际调用 `Run()`。

#### 第三层：降级策略

```go
type DegradationPolicy string

const (
    DegradeFailOpen   DegradationPolicy = "fail_open"   // 返回空结果，不报错
    DegradeFailClosed DegradationPolicy = "fail_closed" // 传播错误（默认）
    DegradeFallback   DegradationPolicy = "fallback"    // 路由到降级插件
    DegradeStaleCache DegradationPolicy = "stale_cache" // 返回上一次成功结果
)
```

#### 配置示例（`plugins/config.yaml`）

```yaml
plugins:
  - name: ai
    enabled: true
    config:
      llm_api_key: sk-xxx
    retry:
      max_retries: 2
      backoff_base: 500ms
    circuit_breaker:
      failure_threshold: 3
      cooldown_period: 60s
    degradation: fallback
    fallback_plugin: ai-cached  # 失败时路由到返回缓存结果的降级插件
```

#### 依赖级联降级

当插件 A 熔断时，依赖 A 的插件 B 自动进入降级模式，不等待自己的调用也失败。

## 6. 多语言插件支持

### 现状

`model.Plugin` 是 Go interface，所有插件必须是编译进二进制或通过 `plugin.Open` 加载的 Go 代码。

### 核心思路

定义一个 **语言无关的插件协议**。Manager 不关心插件用什么语言写，只关心它是否实现了这个协议。与第1点（热加载）和第3点（进程隔离）天然统一。

### 协议设计 — JSON-RPC over stdin/stdout

```
┌──────────────┐    stdin (JSON Line)    ┌──────────────────┐
│              │ ◄────────────────────── │                  │
│ PluginManager│                         │ 任何语言的插件进程 │
│              │ ──────────────────────► │                  │
│  (Go 主进程)  │    stdout (JSON Line)   │  (Python/Node/...) │
└──────────────┘                         └──────────────────┘
```

### 协议消息格式（每行一个完整 JSON）

```json
// Manager → Plugin（请求）
{"id":"1","method":"run","params":{"data":{"messages":[{"role":"user","content":"你好"}]}}}

// Plugin → Manager（成功响应）
{"id":"1","result":{"output":{"response":"你好！有什么可以帮你的？"}}}

// Plugin → Manager（错误响应）
{"id":"1","error":{"code":-1,"message":"api timeout"}}

// Manager → Plugin（健康检查）
{"id":"2","method":"ping"}

// Plugin → Manager
{"id":"2","result":{}}

// Manager → Plugin（配置下发）
{"id":"3","method":"configure","params":{"config":{"llm_api_key":"sk-xxx"}}}

// Plugin → Manager
{"id":"3","result":{}}
```

### Go 侧适配器实现

```go
// handler/external.go（新文件）
type ExternalPlugin struct {
    name    string
    version string
    cmd     *exec.Cmd
    stdin   io.WriteCloser
    stdout  *bufio.Scanner
    mu      sync.Mutex
    pending map[string]chan jsonrpcResponse
    reqID   atomic.Int64
}

// 实现 model.Plugin 接口
func (p *ExternalPlugin) Name() string    { return p.name }
func (p *ExternalPlugin) Version() string { return p.version }

func (p *ExternalPlugin) Run(data map[string]interface{}) (map[string]interface{}, error) {
    // 构造 JSON-RPC 请求 → 写 stdin → 等 stdout 响应 → 返回
}
```

### 插件清单（每个插件目录下一个 `plugin.yaml`）

```yaml
# plugins/my-python-ai/plugin.yaml
name: python-ai
version: 1.0.0
runtime: python3
entrypoint: main.py
protocol: jsonrpc
timeout: 30s
dependencies:
  - name: calculator
    constraint: ">=1.0.0"
```

### 多语言 SDK（极薄封装）

每种语言只需几百行代码实现 JSON-RPC 通信循环：

```
plugins/
├── go-ai/          # Go 原生插件（现有模式）
│   └── ai.go
├── python-ai/      # Python 插件（新）
│   ├── plugin.yaml
│   └── main.py     # 从 pluginmanager-sdk 导入, 装饰 @plugin.run
├── node-tts/       # Node.js 插件（新）
│   ├── plugin.yaml
│   └── main.js
└── calculator/     # Go 原生（现有模式）
    └── calculator.go
```

### Python SDK 使用示例

```python
# main.py
from pluginmanager_sdk import Plugin

plugin = Plugin("python-ai", "1.0.0")

@plugin.run
def handle(data):
    messages = data["messages"]
    response = call_llm_api(messages)  # 插件作者只需关心业务逻辑
    return {"response": response}

plugin.serve()  # 阻塞，处理 stdin/stdout JSON-RPC 循环
```

## 7. 六个特性之间的关系

```
        ┌──────────────────┐
        │  多语言协议 (#6)  │ ◄── 定义了统一的外部插件通信协议
        └────────┬─────────┘
                 │ 依赖
        ┌────────▼─────────┐
        │  热加载/卸载 (#1) │ ◄── 基于协议，启动/停止子进程
        └────────┬─────────┘
                 │ 依赖
        ┌────────▼─────────┐
        │  进程隔离 (#3)   │ ◄── 子进程天然隔离，cgroups 限制资源
        └────────┬─────────┘
                 │ 需要
        ┌────────▼─────────┐
        │  超时控制 (#2)   │ ◄── 超时后 SIGKILL 子进程，彻底清理
        └────────┬─────────┘
                 │ 需要
        ┌────────▼─────────┐
        │  失败降级 (#5)   │ ◄── 超时/崩溃/熔断后的恢复策略
        └────────┬─────────┘
                 │ 编排
        ┌────────▼─────────┐
        │  依赖管理 (#4)   │ ◄── 决定启动顺序，级联降级
        └──────────────────┘
```

### 建议的实现顺序

| 顺序 | 特性 | 理由 |
|------|------|------|
| 1 | 依赖管理 | 纯逻辑层改动，不涉及进程模型，收益高风险低 |
| 2 | 超时控制 | 在现有 goroutine 模型下就能做 |
| 3 | 降级策略 | 依赖超时检测结果，组合成完整容错体系 |
| 4 | 多语言协议 | 最大的架构跃迁，后续热加载和进程隔离都依赖它 |
| 5 | 热加载 | 依赖协议 + 进程管理 |
| 6 | 进程隔离 | 依赖协议 + 超时 + 降级的组合才能完整运作 |
